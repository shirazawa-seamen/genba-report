## 追加機能の詳細設計メモ

### A3: 現場ドキュメント管理

**目的**: 図面・仕様書・工程表を現場でモバイル閲覧可能にする

**実装イメージ**:
- `site_documents` テーブル新設（site_id, type, file_url, file_name, uploaded_by, uploaded_at）
- type: `blueprint`（図面）/ `specification`（仕様書）/ `schedule`（工程表）/ `purchase_order`（発注書）/ `other`
- Supabase Storage に `site-documents` バケット作成
- 現場詳細ページにドキュメントセクション追加
- PDF/画像のプレビュー表示

### A4: 現場セットアップチェック

**目的**: 現場に必要書類が揃っているか一目で確認

**実装イメージ**:
- sites テーブルに以下のフラグ追加:
  - `has_blueprint` (boolean)
  - `has_specification` (boolean)
  - `has_purchase_order` (boolean)
  - `has_schedule` (boolean)
- 現場詳細ページにチェック状況を表示
- ドキュメントアップロード時に自動で true に更新

### A5: 品質確認チェックリスト（3段階）

**目的**: メーカー施工要領書に基づく品質チェックを3段階で管理

**既存**: `process_checklists` テーブル（migration_v3で作成済み）
**拡張**: inspection_phase カラム追加

| 検査段階 | 内容 |
|---------|------|
| `acceptance` | 受け入れ検査：材質が施工書通りか、傷がないか等 |
| `during` | 施工中検査：配管勾配、金具ピッチ等の規定確認 |
| `post` | 施工後検査：仕上がり品質の確認 |

### A6: メール通知

**目的**: 報告提出・承認・差戻し時に関係者へ自動通知

**実装イメージ**:
- Supabase Edge Functions or Resend API
- 通知タイミング:
  - 報告提出時 → 管理者へ通知
  - 管理者承認時 → 元請けへ通知
  - 差戻し時 → 報告者へ通知（修正指示コメント付き）

### A7: 使用材料の報告

**目的**: 使用した材料の品番・数量を記録し、仕様書との整合性を担保

**実装イメージ**:
- `report_materials` テーブル新設（report_id, material_name, product_number, quantity, unit, specification_match）
- 日次報告フォームに「使用材料」セクション追加（Step 2 に組込み）
- 品番入力 → 仕様書の品番との照合表示

## 優先度と実装順序（推奨）

3/16運用開始に向けて、以下の順序で実装を推奨:

```
Week 1（2/24〜3/2）: ロール・認証基盤
  ├─ 4ロール対応（DB migration + RLS）
  ├─ ロール別UI出し分け
  └─ アカウント招待機能

Week 2（3/3〜3/9）: 承認フロー + ドキュメント管理
  ├─ 承認・差戻しフロー
  ├─ 現場ドキュメント管理（A3）
  ├─ 現場セットアップチェック（A4）
  └─ デモ用サンプルデータ投入（B2）

Week 3（3/10〜3/16）: 品質・通知・仕上げ
  ├─ 品質確認チェックリスト 3段階（A5）
  ├─ メール通知（A6）
  ├─ 使用材料の報告（A7）
  └─ バグ修正・テスト・運用準備
```

## 出来高請求に関する方針メモ

議事録での結論:
- 小規模工事では中間請求（50%請求等）は一般的でなく、0% or 100%が基本
- 進捗率はあくまで目安。実際の完了判断は元請けが写真で確認
- **現行の進捗率（%）入力はそのまま維持**。図面ベースの視覚的追跡（C3）はPhase 2で検討
