# 建築事業部 現場報告システム - 要件定義書

## 1. システム概要

### 1.1 目的
建築事業部における現場作業の透明化と標準化を実現し、以下を達成する:
- 県外企業からの信頼獲得(「見える工事®」の実現)
- 現場作業の属人化解消
- 報告業務の効率化と品質向上
- 元請けとのコミュニケーション強化

### 1.2 システムの位置づけ
- **透明化施策**の中核システム
- **作業標準化・マニュアル化**を支援するツール
- **Notionデータベース**との連携を想定

---

## 2. ユーザー定義

### 2.1 主要ユーザー
1. **現場作業者(職人)**
   - 日次報告の入力
   - 写真・動画のアップロード
   - 作業マニュアルの参照

2. **現場監督(施工管理者)**
   - 進捗管理
   - 報告内容の確認・承認
   - 元請けへの報告資料作成

3. **元請け企業**
   - 進捗状況の閲覧
   - 報告書の受領
   - 問題点の確認
   - 品質管理

---

## 3. 機能要件

### 3.1 現場報告機能(コア機能)

#### 3.1.1 日次報告
**必須項目:**
- 現場名
- 報告日
- 作業工程
- 作業内容(テキスト)
- 作業者名(複数選択可)
- 進捗率(%)
- 写真(最低4枚:現場四隅)
- 作業前/作業後の写真

**任意項目:**
- 動画(30秒程度の短編)
- 問題点・遅延理由
- 改善提案
- 天候
- 作業時間

#### 3.1.2 週次報告
- 日次報告の自動集計
- 定点撮影写真の時系列表示
- 進捗率の推移グラフ
- 週間サマリー(自動生成)

#### 3.1.3 写真・動画管理
- **撮影ガイドライン表示**
  - 角度・光・枚数の指示
  - 撮影例の表示
- **自動分類**
  - 工程別
  - 日付別
  - 作業者別
- **ファイル命名規則の自動適用**
  - 形式: `{現場名}_{工程}_{日付}_{連番}.jpg`
- **定点撮影管理**
  - 四隅の位置記録
  - 同一アングル確認機能

### 3.2 工程管理機能

#### 3.2.1 工程表管理
- 工程表の作成・編集
- ガントチャート表示
- 進捗の可視化
- 遅延アラート

#### 3.2.2 役割分担管理
- 朝礼での役割分担入力
- 作業者別タスク表示
- 実績時間の記録

### 3.3 マニュアル・ナレッジ機能

#### 3.3.1 作業マニュアル参照
- 工程別マニュアル表示
- 動画マニュアル再生
- 写真付き手順書
- 注意点・こだわりポイント表示

#### 3.3.2 現場ルール表示
- 喫煙ルール
- 清掃ルール
- 挨拶ルール
- ゴミ分別ルール
- 写真撮影ルール

### 3.4 コミュニケーション機能

#### 3.4.1 元請けへの報告
- 報告書の自動生成(PDF)
- LINE/Chatwork/Slack連携
- メール送信
- 報告履歴管理

#### 3.4.2 社内コミュニケーション
- コメント機能
- 問題点の共有
- 改善提案の投稿

### 3.5 分析・レポート機能

#### 3.5.1 ダッシュボード
- 全現場の進捗一覧
- 遅延現場の抽出
- 作業者別稼働状況
- 工程別所要時間分析

#### 3.5.2 レポート出力
- 週次報告書(自動生成)
- 月次報告書
- 工程別実績レポート
- Before/Afterアルバム

### 3.6 CCUS連携機能

#### 3.6.1 技能者管理
**基本機能:**
- 技能者のCCUS ID登録・管理
- CCUS登録情報の取得
  - 氏名、生年月日
  - 保有資格
  - 技能者レベル(1〜4)
  - 就業日数
- 技能者マスタとの紐付け

**表示機能:**
- 現場配置画面での資格表示
- レベル別の技能者一覧
- レベルアップ進捗の可視化

#### 3.6.2 就業履歴の自動蓄積
**自動送信データ:**
- 現場情報
  - 工事名、場所、期間
  - 元請け情報
- 技能者の就業情報
  - 就業日
  - 職種(作業工程から自動判定)
  - 立場(職長/作業員)
  - 作業時間
- 施工体制情報
  - 元請け・下請け構造
  - 各社の役割

**送信タイミング:**
- 日次報告の承認時に自動送信
- 手動での一括送信機能
- 送信エラー時の再送機能

#### 3.6.3 施工体制情報の連携
- 現場ごとの施工体制図作成
- 元請け・下請け関係の登録
- CCUSへの施工体制情報送信
- 登録事業者率の表示

#### 3.6.4 入退場管理連携(将来対応)
- QRコード/カードリーダーとの連携
- 入退場時刻の自動記録
- CCUSへの入退場データ送信
- 就業履歴との自動紐付け

#### 3.6.5 統計・分析
- CCUS活用状況ダッシュボード
  - 登録技能者率
  - 就業履歴蓄積率
  - 技能者レベル分布
- 公共工事加点対応レポート
- 技能者の成長レポート

**CCUS連携の要件:**
> 本システムは「就業履歴データ登録標準API連携認定システム」として、一般財団法人建設業振興基金の認定取得を目指します。

**認定要件:**
- 標準API仕様への準拠
- 就業履歴情報の真正性保証
- セキュリティ基準の遵守
- データの正確性・信頼性の確保

## 4. 非機能要件

### 4.1 ユーザビリティ
- **スマホ最適化必須**
  - 現場での入力を想定
  - タッチ操作に最適化
  - オフライン対応(後で同期)
- **ITリテラシーが低い職人でも使える**
  - 直感的なUI
  - 最小限の入力項目
  - 音声入力対応
  - 写真は撮影→即アップロード

### 4.2 パフォーマンス
- 写真アップロード: 10秒以内
- ページ読み込み: 3秒以内
- 動画アップロード: バックグラウンド処理

### 4.3 セキュリティ
- ユーザー認証(ログイン)
- 権限管理(役割別)
- データバックアップ(日次)
- SSL/TLS通信

### 4.4 拡張性
- Notion API連携
- 外部カレンダー連携
- 会計システム連携(将来)

## 5. 技術要件

### 5.1 推奨技術スタック
- **フロントエンド**: React / Next.js
- **バックエンド**: Node.js / Express
- **データベース**: PostgreSQL / Supabase
- **ストレージ**: AWS S3 / Cloudflare R2
- **認証**: Auth0 / Supabase Auth
- **ホスティング**: Vercel / AWS App Runner

### 5.2 外部連携
- **CCUS API**(建設キャリアアップシステム - 就業履歴データ登録標準API)
- Notion API
- LINE Messaging API
- Google Maps API(現場位置)

## 9. リスクと対策

### 9.1 リスク
1. **職人のITリテラシー不足**
   - 対策: 徹底的なUI簡素化、動画マニュアル、個別トレーニング

2. **現場でのネット環境不安定**
   - 対策: オフライン対応、自動同期機能

3. **入力の手間による抵抗**
   - 対策: 音声入力、テンプレート、最小限の必須項目

4. **既存ツール(ANDPAD)との競合**
   - 対策: 一括発注の強みを活かした独自機能、使いやすさでの差別化
